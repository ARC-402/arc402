### AUD-4337-01

**ID:** AUD-4337-01
**Severity:** Critical
**Title:** Governance functions use `onlyOwner` — ERC-4337 governance UserOps always revert at execution

**Description:**
`validateUserOp` correctly classifies and validates governance operations (setGuardian, updatePolicy, proposeRegistryUpdate, setVelocityLimit, freeze(string), unfreeze) — returning `SIG_VALIDATION_SUCCESS` when the owner's signature over `userOpHash` is valid.

However, when the EntryPoint subsequently calls the function directly (`sender.call(userOp.callData)`), `msg.sender` is `address(entryPoint)`, not `owner`. All six governance functions used the `onlyOwner` modifier which checks `msg.sender == owner` — causing every governance UserOp to revert with `WAuth()` at execution, regardless of how valid the signature was.

**Result:** The entire ERC-4337 governance pathway is non-functional. Governance can only be exercised by direct owner EOA calls; the "cold master key signs a governance UserOp" flow described in Spec 30 does not work.

**PoC:**
```solidity
// 1. Owner signs a governance UserOp for setGuardian
bytes memory sig = ownerSign(userOpHash);
PackedUserOperation memory userOp = buildUserOp(
    abi.encodeWithSignature("setGuardian(address)", newGuardian),
    sig
);

// 2. EntryPoint calls validateUserOp — returns 0 (SIG_VALIDATION_SUCCESS) ✓
vm.prank(entryPoint);
uint256 result = wallet.validateUserOp(userOp, userOpHash, 0);
assert(result == 0);

// 3. EntryPoint calls the function — REVERTS WAuth() ✗
vm.prank(entryPoint);
wallet.setGuardian(newGuardian); // reverts: msg.sender = entryPoint ≠ owner
```

**Fix:**
Change `onlyOwner` to `onlyEntryPointOrOwner` on all governance functions that appear in `_isGovernanceOp`. The security gate is `validateUserOp` — the EntryPoint is trusted to call governance functions only after owner signature verification passes.

Functions changed:
- `setGuardian`
- `updatePolicy`
- `proposeRegistryUpdate`
- `cancelRegistryUpdate` *(also AUD-4337-02)*
- `executeRegistryUpdate` *(also AUD-4337-02)*
- `setAuthorizedInterceptor` *(also AUD-4337-02)*
- `setVelocityLimit`
- `freeze(string calldata)` — owner-initiated freeze
- `unfreeze`

**Diff:**
```solidity
// Before
function setGuardian(address _guardian) external onlyOwner {
function updatePolicy(bytes32 newPolicyId) external onlyOwner {
function proposeRegistryUpdate(address newRegistry) external onlyOwner {
function cancelRegistryUpdate() external onlyOwner {
function executeRegistryUpdate() external onlyOwner {
function setAuthorizedInterceptor(address interceptor) external onlyOwner {
function setVelocityLimit(uint256 limit) external onlyOwner {
function freeze(string calldata reason) external onlyOwner {
function unfreeze() external onlyOwner {

// After
function setGuardian(address _guardian) external onlyEntryPointOrOwner {
function updatePolicy(bytes32 newPolicyId) external onlyEntryPointOrOwner {
function proposeRegistryUpdate(address newRegistry) external onlyEntryPointOrOwner {
function cancelRegistryUpdate() external onlyEntryPointOrOwner {
function executeRegistryUpdate() external onlyEntryPointOrOwner {
function setAuthorizedInterceptor(address interceptor) external onlyEntryPointOrOwner {
function setVelocityLimit(uint256 limit) external onlyEntryPointOrOwner {
function freeze(string calldata reason) external onlyEntryPointOrOwner {
function unfreeze() external onlyEntryPointOrOwner {
```

### AUD-4337-02

**ID:** AUD-4337-02
**Severity:** High
**Title:** `cancelRegistryUpdate`, `executeRegistryUpdate`, `setAuthorizedInterceptor` absent from `_isGovernanceOp` — auto-approve without owner signature after AUD-4337-01 fix

**Description:**
Three governance-sensitive functions are missing from `_isGovernanceOp`:

| Function | Impact if auto-approved |
|---|---|
| `cancelRegistryUpdate` | Anyone can cancel a pending registry upgrade |
| `executeRegistryUpdate` | Anyone can complete a registry upgrade (subject to timelock, but still unauthorized) |
| `setAuthorizedInterceptor` | Anyone can replace the interceptor — the interceptor has `executeTokenSpend` access, enabling token drain |

Before the AUD-4337-01 fix, these functions failed at execution anyway (`onlyOwner` blocked EntryPoint). But after fixing AUD-4337-01 (`onlyEntryPointOrOwner`), an unsigned protocol UserOp targeting any of these functions would:
1. Pass `validateUserOp` as a protocol op (auto-approve, frozen check only)
2. Execute successfully via EntryPoint

`setAuthorizedInterceptor` is the highest-risk omission: the authorized interceptor can call `executeTokenSpend` to transfer any ERC-20 token from the wallet. Replacing it with a malicious address would enable complete token drain.

**PoC (post AUD-4337-01 fix, pre AUD-4337-02 fix):**
```solidity
// Attacker submits a UserOp with NO signature
PackedUserOperation memory userOp = buildUserOp(
    abi.encodeWithSignature("setAuthorizedInterceptor(address)", attackerDrain),
    ""  // no signature
);
// validateUserOp: not governance op → auto-approve → returns 0
vm.prank(entryPoint);
uint256 result = wallet.validateUserOp(userOp, hash, 0);
assert(result == 0);  // passes!

// EntryPoint executes — interceptor is now attacker's drain contract
vm.prank(entryPoint);
wallet.setAuthorizedInterceptor(attackerDrain);  // succeeds!
```

**Fix:**
Add all three to `_isGovernanceOp`:

```solidity
function _isGovernanceOp(bytes4 selector) internal pure returns (bool) {
    return selector == this.setGuardian.selector
        || selector == this.updatePolicy.selector
        || selector == this.proposeRegistryUpdate.selector
        || selector == this.cancelRegistryUpdate.selector      // added
        || selector == this.executeRegistryUpdate.selector     // added
        || selector == this.setAuthorizedInterceptor.selector  // added
        || selector == this.setVelocityLimit.selector
        || selector == bytes4(keccak256("freeze(string)"))
        || selector == this.unfreeze.selector;
}
```

### AUD-4337-03

**ID:** AUD-4337-03
**Severity:** Medium
**Title:** `_validatePolicyBounds` only checks frozen state — bundler gas-waste / griefing vector

**Description:**
Protocol ops auto-approve based solely on `frozen` state:

```solidity
function _validatePolicyBounds() internal view returns (uint256) {
    return frozen ? SIG_VALIDATION_FAILED : SIG_VALIDATION_SUCCESS;
}
```

A protocol UserOp targeting `executeSpend`, `executeTokenSpend`, or `executeContractCall` will validate successfully even if it will revert at execution (e.g., context not open, attestation invalid, policy limit exceeded). The bundler pays gas for simulation and inclusion of the op, then the execution reverts. Bundlers that don't perform deep simulation can be griefed into wasting gas.

**Impact:** DoS / gas griefing against bundler operators. Not a fund-security issue — the execution-time checks correctly prevent unauthorized fund movement.

**PoC:**
```solidity
// Wallet not frozen, context NOT open
// Submit executeSpend UserOp — validates OK, reverts at execution
PackedUserOperation memory userOp = buildUserOp(
    abi.encodeWithSignature("executeSpend(address,uint256,string,bytes32)", ...),
    ""
);
vm.prank(entryPoint);
uint256 result = wallet.validateUserOp(userOp, hash, 0);
assert(result == 0);  // passes — but will revert at execution (WCtx)
```

**Fix (not implemented — Medium, no fund risk):**
Add a lightweight pre-flight check inside `_validatePolicyBounds` that decodes the selector and gates on `contextOpen` for spend ops. Alternatively, document in bundler integration guide that bundlers MUST simulate execution (not just call `validateUserOp`) before inclusion.

### AUD-4337-04

**ID:** AUD-4337-04
**Severity:** Low
**Title:** `validateUserOp` not `nonReentrant` — potential ETH double-drain on prefund via malicious EntryPoint

**Description:**
`validateUserOp` sends ETH to the EntryPoint near the end of the function:

```solidity
if (missingAccountFunds > 0) {
    (bool ok,) = payable(address(entryPoint)).call{value: missingAccountFunds}("");
    if (!ok) revert WPrefund();
}
```

A malicious or compromised EntryPoint could re-enter `validateUserOp` during this call. Since `msg.sender == address(entryPoint)` passes, reentrancy would succeed. The re-entered call could again transfer `missingAccountFunds`, doubling the ETH drain.

**Mitigating factors:**
- `entryPoint` is immutable — requires deploying the wallet against a malicious EntryPoint from the start.
- No meaningful state change between re-entrant calls that could be exploited in other ways.

**Fix (not implemented — Low, requires malicious EP from genesis):**
Add `nonReentrant` modifier to `validateUserOp`, or move the prefund transfer before the validation logic (checks-effects-interactions).

### AUD-4337-05

**ID:** AUD-4337-05
**Severity:** Info
**Title:** No P256/secp256r1 signature path — ECDSA only

**Description:**
`_validateOwnerSignature` exclusively supports secp256k1 ECDSA via `ECDSA.tryRecover`. Hardware security keys (YubiKey, Apple Secure Enclave, Android Keystore) and WebAuthn/passkey signers use secp256r1 (P256). A wallet owner using a hardware-backed passkey as the master key cannot sign governance UserOps.

**Fix (not implemented — Info, future feature):**
Add a P256 signature path using EIP-7212 (RIP-7212 precompile on Base) or an on-chain verifier (e.g., `P256Verifier`). The signature format should include a 1-byte prefix indicating ECDSA vs P256.

### AUD-4337-06

**ID:** AUD-4337-06
**Severity:** Info
**Title:** Inline `bytes4(keccak256("freeze(string)"))` for overloaded selector — correct but fragile

**Description:**
`_isGovernanceOp` uses an inline hash for the overloaded `freeze` function:

```solidity
|| selector == bytes4(keccak256("freeze(string)"))   // owner freeze (overloaded)
```

This is correct — `this.freeze.selector` would be ambiguous due to overloading. However, the inline string is not verified by the compiler. A future rename, parameter type change, or copy-paste error would silently misclassify governance ops.

**Fix (not implemented — Info):**
Define a compile-time constant:
```solidity
bytes4 internal constant FREEZE_WITH_REASON_SELECTOR = bytes4(keccak256("freeze(string)"));
```
And add a Foundry invariant test that verifies the selector matches `abi.encodeWithSignature("freeze(string)", "")[:4]`.
