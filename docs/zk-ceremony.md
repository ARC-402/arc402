# ARC-402 ZK Trusted Setup Ceremony

## What This Is

ARC-402 includes three ZK privacy extensions:

| Extension | What It Proves |
|-----------|---------------|
| **ZK Trust Threshold** | "My trust score ≥ X" without revealing the exact score |
| **ZK Solvency** | "I can cover this escrow" without revealing total balance |
| **ZK Capability** | "I have this capability" without revealing full capability set |

The gate contracts, verifier contracts, and Circom circuits are complete. The ZK layer is architecturally ready.

**One thing is missing: the trusted setup ceremony.**

---

## Why a Ceremony Is Required

Groth16 ZK proofs rely on a set of cryptographic parameters called the Proving Key and Verification Key. These parameters are generated from random toxic waste — secret randomness that must be destroyed after the ceremony.

If even one participant honestly destroys their toxic waste, the system is secure. This is why the ceremony requires multiple independent participants. No single party needs to be trusted. The security assumption is: "at least one of these N participants was honest."

The contracts currently contain development keys generated by a single party. These are insecure for mainnet — anyone who knows the toxic waste can forge proofs. The ceremony replaces them with keys generated by the community.

---

## How the Ceremony Works

### Phase 1 — Powers of Tau (universal)
The underlying elliptic curve parameters. ARC-402 will use an existing Powers of Tau ceremony (Hermez or Zcash) — this phase is already complete and does not need to be repeated.

### Phase 2 — Circuit-specific setup (this ceremony)
Each of the three circuits needs its own Phase 2 ceremony:

1. `trust-threshold.circom` — range proof (~50 constraints)
2. `solvency.circom` — range + Merkle proof (~200 constraints)
3. `capability.circom` — set membership proof (~150 constraints)

Each participant:
1. Downloads the previous participant's `.zkey` contribution file
2. Contributes their own randomness: `snarkjs zkey contribute prev.zkey next.zkey`
3. Publishes their contribution publicly
4. Destroys their randomness

The final `.zkey` is used to export the Solidity verifier contract, which replaces the dev key contracts.

## How to Participate

You need: Node.js, snarkjs, and about 10 minutes.

```bash
npm install -g snarkjs

# Download the current contribution file from the ceremony coordinator
curl -O https://ceremony.arc402.xyz/trust-threshold-current.zkey

# Contribute your randomness
snarkjs zkey contribute trust-threshold-current.zkey trust-threshold-my-contribution.zkey \
  --name="Your Name or Handle" \
  -v

# Submit your contribution
curl -X POST https://ceremony.arc402.xyz/contribute \
  -F "file=@trust-threshold-my-contribution.zkey" \
  -F "name=Your Name or Handle" \
  -F "circuit=trust-threshold"

# Destroy your local copy of both files
rm trust-threshold-current.zkey trust-threshold-my-contribution.zkey
```

Repeat for `solvency` and `capability` circuits.

## What Happens After the Ceremony

Once we have contributions from at least 10 independent participants:

1. The ceremony coordinator exports the final verification keys
2. New Solidity verifier contracts are generated: `snarkjs generateverifier`
3. Contracts are deployed to Base mainnet
4. ZKTrustGate, ZKSolvencyGate, ZKCapabilityGate are pointed at the new verifiers
5. PolicyEngine integration is activated
6. ZK mode goes live — opt-in for all agents

## What Gets Unlocked

**For agents:**
- Prove trust eligibility to counterparties without revealing exact score
- Prove solvency for high-value agreements without revealing balance
- Prove capability for competitive work without disclosing full skill set

**For the protocol:**
- Mature marketplace dynamics — reputation becomes a strategic asset, not just a public number
- Enterprise adoption path — firms that need confidentiality can participate without full transparency
- Competitive privacy — agents in high-value niches can operate without revealing their positioning

## Why This Is a Community Moment

The trusted setup ceremony is the one part of ARC-402 that cannot be done alone. The code is written. The circuits are verified. The contracts are deployed.

What remains is participation.

Every contributor to the ceremony is permanently part of the protocol's cryptographic foundation. Their contribution is recorded in the final `.zkey` file. As long as one contributor was honest, the privacy guarantees hold for every agent who uses ZK mode — forever.

This is how cryptographic trust is built: not by a single authority, but by participation.
